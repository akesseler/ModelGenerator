Adjustment

The settings of Adjustment are used to control how classes and their members (properties and/or fields) are named during code generation. All possible options are shown below.

Enable usage of adjustment feature.
Any additional class and member name modification is skipped in case of this checkbox is unchecked. Otherwise automatic class and member name adjustment takes place.
Casing Format
This selection box allows to choose how class and member renaming should be applied. Possible values are shown below.
Unmodified A renaming of classes and members does not take place and each descriptor remains unchanged.
PascalCase The naming convention for Pascal Case is applied. This means that the first letter of each class and member name (property or field) is converted into upper case.
CamelCase The naming convention for Camel Case is applied. This means that the first letter of each class and member name (property or field) is converted into lower case.
Member Prefix
A string to be prepended to each member name (property or field). A prefix is actually only useful for fields. Please note that each prefix is only applied after the naming convention rules have been applied. Furthermore, please be aware that a prefix is applied as it is, including special characters, whitespaces, etc.
Member Suffix
A string to be appended to each member name (property or field). A suffix is actually only useful for fields. Please note that each suffix is only applied after the naming convention rules have been applied. Furthermore, please be aware that a suffix is applied as it is, including special characters, whitespaces, etc.

Replacements

Value replacement takes place during model import and is necessary to ensure type dependent properties and fields. In addition, the procedure of value replacement is case sensitive and replaces each fitting word.

Put each value to be replaced that can occur in a source model (JSON or XML) into the left column and put the replacement into its corresponding right column cell.

The right cell should contain a typed value, for example strings with double quotes, integer numbers with just zero, floating point number with zero–dot–zero, date and/or time values with a particular date and/or time, and so on.

Example

The source model may contain the literal “integer”. But this literal cannot be converted into any valid type, except string. Therefore, each literal “integer” should be replaced by “0”, which indeed indicates an integer data type.

Another example is the literal “number” that may represent a decimal value. In such a case it would be useful to replace each “number” by “0.0” instead.
